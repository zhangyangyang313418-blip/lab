核心业务流程

产品测试平台的流程建议按三栏结构理解：左侧是角色和触发条件，中间是主流程，右侧是输入、输出和系统模块。系统分流应早于审批，资源确认应在审批后发生。

PHASE 01
测试申请
申请人提交项目与测试需求录入项目、产品、平台、样件、测试目标、变更描述和附件材料。
输出测试申请单、项目基础数据、附件索引。
PHASE 02
Test Flow 形成
系统形成结构化测试流按平台模板、变更类型和测试属性生成可计算的 Test Flow，而不是只生成普通文档。
模块环境大纲 seed、平台模板、测试项目配置。
PHASE 03
系统筛选与分流
判断内部 / 委外 / 混合路径根据能力、设备、费用归属和特殊规则决定执行路径。
分支内部测试、委外测试、内部 + 委外混合测试。
PHASE 04
测试申请审核
审批测试内容和费用口径确认测试项目、样品范围、费用规则、报价依据和交付模板。
异常退回修改、补充资料、重新提交。
PHASE 05
资源确认
确认设备、人力、排期和委外资源内部资源不足时返回排期反馈，必要时转委外或混合执行。
关联实验室资源看板、软锁定、硬锁定、委外报价。
PHASE 06
测试执行
按 Test Flow 执行并记录过程内部与委外路径分别保留测试记录、异常、过程附件和费用变化。
输出测试记录、过程日志、费用明细、问题项。
PHASE 07
报告确认
确认报告内容和数据一致性内部报告与委外报告分别确认，必要时补充复测或报告修订。
检查测试标准、样件编号、结果、附件、签核状态。
PHASE 08
报告归档
归档全链路材料归档申请、项目、产品、标准、Test Flow、测试记录、附件、过程日志和正式报告。
沉淀可追溯项目包和后续复用规则。

风险与维护原则

这个项目的风险主要来自多来源规则不同步：代码、Excel、页面、交接文档和用户本地草稿任何一处落后，都会造成看似“改了但没生效”的问题。
